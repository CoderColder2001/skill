# `code-plan` Skill Design

## 概要

`code-plan` 是一个显式触发的编码前规划 skill。它只在用户明确输入 `/code-plan` 后进入工作流，并要求 agent 在真正开始生产代码改动之前，先把本次实现任务整理成一份落在目标项目里的 task-scoped spec。

它的定位不是“大而全的架构分析”，而是编码前的一道轻量工程闸门：让本次实现的目标、影响面、模块边界和新增模块决策先被说清楚。

## 设计背景

当前已有 skill 生态已经覆盖了几个相邻环节：

- `brainstorming`：偏通用的设计澄清和需求探索
- `writing-plans`：把已确认 spec 继续细化成实现计划
- `requesting-code-review` / `receiving-code-review`：偏实现后的 review 环节
- `code-analyse`：偏显式触发、文档优先的模块/仓库分析

缺少的是一个更贴近“马上要改代码”这一刻的 skill：

- 要显式触发
- 要落文档到目标项目里
- 要比完整架构分析轻
- 但又要比一句“我准备这么改”更有约束力

`code-plan` 就是补这个空位。

## 目标

- skill 名称为 `code-plan`
- 触发方式固定为 `/code-plan`
- 只服务于真实的实现请求，不扩张成通用分析流程
- 在目标项目内生成 `code-plan-docs/`
- 在开始编码前先完成 spec
- spec 结构固定，适合工程实现前审视
- 新增模块时必须给出明确的架构陈述

## 非目标

- 不替代产品级或需求级 brainstorming
- 不替代详细 implementation plan
- 不替代 TDD
- 不替代代码写完后的 review
- 不为了小范围改动强制升级成全仓库架构文档

## 触发模型

- 只有在用户显式输入 `/code-plan` 时才触发
- 对普通编码请求不自动生效
- 一旦进入 `/code-plan` 模式，当前实现序列都应遵守“先 spec，后实现”
- 如果用户后续只是继续讨论概念，不急着改代码，可以先回答，不强制立即落 spec

## 输出模型

文档写到被操作项目，而不是 skill 自己的目录里。

默认结构：

```text
code-plan-docs/
  current-spec.md
  specs/
    YYYY-MM-DD-<topic>.md
```

说明：

- `specs/YYYY-MM-DD-<topic>.md`：本次任务的正式 spec
- `current-spec.md`：当前任务的稳定入口
- 如果项目里已经有 `code-plan-docs/`，应增量更新，而不是再造一套并行目录

## 范围控制

`code-plan` 应明显轻于 `code-analyse`：

- 只看和当前实现任务直接相关的模块、接口、调用关系、测试和配置
- 只有在正确性、公共接口、数据归属或新增模块放置位置受影响时，才扩大范围
- 不把小改动默默扩成仓库级分析
- 如果证据不足，应在 spec 里显式写出不确定项，而不是补脑

## 工作流

1. 判断对话是否已经从“讨论”进入“准备实现”
2. 确认目标项目根目录和任务 slug
3. 阅读相关代码、测试、配置和模块边界
4. 生成 `code-plan-docs/specs/YYYY-MM-DD-<topic>.md`
5. 刷新 `code-plan-docs/current-spec.md`
6. 向用户汇报 spec 摘要并等待确认
7. spec 认可后，才进入真正实现
8. 如果后续任务范围发生实质变化，先更新 spec，再继续改代码

## Spec 结构

每次生成的正式 spec 必须包含这些部分，且顺序固定：

1. `## 当前编码意图`
2. `## 预期目标与完成标准`
3. `## 当前相关模块现状`
4. `## 本次涉及改动模块`
5. `## 新模块架构设计陈述`
6. `## 风险与待确认项`
7. `## 不在本次范围内的内容`

各部分意图如下：

- `当前编码意图`：把用户请求转成工程实现语境
- `预期目标与完成标准`：明确之后可以验证的成功条件
- `当前相关模块现状`：基于已读代码说明当前边界和约束
- `本次涉及改动模块`：说明哪些模块会被触及，以及为什么
- `新模块架构设计陈述`：如果需要新增模块，必须说明职责、边界、依赖方向、建议路径和变更后调用链
- `风险与待确认项`：显式记录不确定性和风险
- `不在本次范围内的内容`：压住 scope creep

## 和现有 skill 的边界

### 相对 `brainstorming`

`code-plan` 更窄，也更工程实现导向。它关注的是“这次具体要怎么改代码”，而不是更宽的产品、需求或交互设计问题。

### 相对 `writing-plans`

`code-plan` 在 spec 被确认后就该停住。如果接下来需要更细的分步执行计划，应交给 `writing-plans`。

### 相对 code review 类 skill

这些 skill 面向实现后阶段。`code-plan` 面向实现前阶段。

### 相对 `code-analyse`

`code-plan` 借用了 `code-analyse` 的几个优点：

- 显式触发
- 读代码后再写文档
- 文档默认中文
- 结果稳定落盘

但它输出更轻、更窄、更贴近当前实现任务。

## 验证方式

当前已经用两类 rehearsal 场景验证过：

- 现有模块内的 bugfix 场景
- 需要新增协调模块的结构性变更场景

重点验证点包括：

- spec 是否真的写进目标项目
- 是否只围绕相关模块展开
- 是否包含固定章节
- 新模块场景下，是否能写清承压旧模块、建议路径和变更后调用链

## 已确定决策

- 触发命令：`/code-plan`
- 默认输出目录：`<target-project>/code-plan-docs/`
- 交互模型：task-scoped spec + `current-spec.md`
