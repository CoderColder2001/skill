# `debug-review` Skill Design

## 概要

`debug-review` 是一个显式触发的 bug 分析 skill。它只在用户明确输入 `/debug-review` 后进入工作流，围绕“用户描述的 bug 表现”而不是“当前 diff”展开审查：定位相关模块、回看关键上下游链路、区分高置信原因和待确认假设，并把结果沉淀成目标仓库内可持续修订的 Markdown 文档。

这个 skill 的默认目标是当前工作仓库。用户显式提供路径时，才切换到其他目标仓库。它默认停在分析阶段，不直接改代码，也不自动生成修改方案；只有当用户明确同意进入“修改方案设计”时，当前线程才切换到 `code-plan` 工作流，并把最新的 bug 分析文档作为上游上下文。

## 设计背景

当前工作区已经有几个相邻但不重合的 skill：

- `code-review`：以当前未提交 diff 为中心，找传播遗漏和不一致
- `code-analyse`：以仓库或模块架构说明为中心，产出长期文档
- `code-plan`：以真正动手改代码前的 task-scoped spec 为中心
- `skill-debug`：给 skill 本身接入结构化 debug/trace 能力

缺少的是一个更贴近真实 bug 排查前半段的 skill：

- 用户先描述现象，而不是先给出 diff
- agent 需要围绕 bug 线索去追模块和关键链路
- 结果应该落成一份可以持续修订的分析文档
- 用户确认分析站得住后，再决定是否进入修改方案设计

`debug-review` 用来补这个空位。

## 目标

- 新增一个名为 `debug-review` 的独立 skill，目录位于 `/Users/bytedance/workspace/skill/debug-review`
- 触发方式固定为显式 `/debug-review`
- 默认分析当前工作仓库；只有用户显式给路径时才切换目标
- 以 bug 表现为起点，审查相关模块、入口、调用链、数据链路、配置和测试线索
- 在目标仓库内写入 `debug-review-docs/`
- 保持 `current-review.md` 作为稳定入口，并保留带时间戳的历史分析快照
- 支持用户通过对话反馈或直接修改文档来补充、订正分析结论
- 默认停在 bug 分析阶段
- 当用户明确同意进入修改方案设计时，切换到 `code-plan`
- 在 `README.md` 中给出全局安装、复制到其他工作区和打包文件的使用说明
- 提供最小可用的 `evals/evals.json`，覆盖 bug 现象分析、文档修订和 handoff 场景

## 非目标

- 不自动修改生产代码
- 不自动执行仓库级的泛化架构分析
- 不替代 `code-review` 的 diff-first 审查
- 不替代 `skill-debug` 的 runtime trace 接入和异步 review 能力
- 不在用户未确认的情况下自动进入 `code-plan`
- 不强行给出没有代码证据支撑的根因结论
- 不把“建议补充验证动作”等同于“已经验证通过”

## 触发模型

### 显式触发

只有在用户明确输入 `/debug-review` 时进入该工作流。

### 工作流保持

一旦进入 `/debug-review`，当前线程在该 bug 分析序列内应继续遵守 `debug-review` 规则，直到发生以下任一情况：

- 用户要求结束该轮分析
- 用户明确切换到其他任务
- 用户明确同意进入 `code-plan`

### 不自动触发

以下场景不自动触发 `debug-review`：

- 普通的 bugfix 编码请求
- 单纯的日志解释或报错释义
- 当前 diff 的遗漏审查
- 宽泛的模块架构说明请求

如果用户只是问一个窄的代码问题，而没有显式进入 `/debug-review`，应直接回答，不切入该工作流。

## 默认目标与范围解析

### 默认目标

- 默认目标是当前工作仓库
- 用户显式给出路径时，使用该路径作为目标仓库

### 起始输入

用户至少应提供以下信息中的一部分：

- bug 表现
- 复现线索
- 相关模块、页面、接口或命令
- 已知现象或怀疑点

如果这些线索不足以定位审查范围，可以先问一个简洁澄清问题，但不要在问题未解决前扩张成仓库级分析。

### 范围控制

`debug-review` 的范围应明显窄于 `code-analyse`，但比 `code-review` 更偏链路分析：

- 从 bug 表现出发定位入口或最近的触发点
- 回看关键调用方、被调方、数据转换、配置、测试或运行时分支
- 只追对当前 bug 判断真正有帮助的上下游链路
- 如果证据不足，应在报告中明确写成待确认假设，而不是补脑成结论

## 输出模型

### 对话内输出

在对话里给用户一个简洁摘要，至少包括：

- 当前审查范围
- 已形成的高置信判断
- 仍待确认的关键假设
- 下一步可选入口：继续补充 / 修订文档 / 进入 `code-plan`

### 目标仓库内输出

默认写入：

```text
debug-review-docs/
  current-review.md
  reviews/
    YYYY-MM-DD-HHmm-<topic>.md
```

规则：

- 每次形成一轮新的分析结论或修订结论时，新建一份 `reviews/YYYY-MM-DD-HHmm-<topic>.md`
- 每次都刷新 `current-review.md`，使其始终指向当前活跃分析
- `<topic>` 从 bug 表现和主链路意图里提炼为简洁的 kebab-case
- 如果目标仓库已存在 `debug-review-docs/`，则增量更新，而不是再创建平行目录

这种写法能同时满足两个目标：

- `current-review.md` 适合继续工作和后续 handoff
- `reviews/*.md` 适合保留用户补充前后的历史分析快照

## 证据与结论模型

所有原因判断都必须分成两个层级。

### 高置信原因

只有同时具备以下条件时，才能写入高置信原因：

1. **现象对齐证据**：代码路径、状态转换或模块行为能够解释当前 bug 表现
2. **源码证据**：当前仓库中存在明确的代码、配置、测试或调用链证据支撑这个判断

高置信原因必须直接写清“是哪些文件或链路支持了这个判断”。

### 待确认假设

当信号存在，但证据还不足以负责任地宣称“根因已确定”时，应写入待确认假设。每个假设都要附上：

- 为什么它值得怀疑
- 当前缺了什么证据
- 最小验证动作是什么

这样可以降低噪音，也让用户知道后续应补什么信息。

## 工作流

1. 解析 `/debug-review` 的目标仓库和 bug 线索
2. 确认默认目标是当前仓库，除非用户显式改路径
3. 读取与 bug 表现直接相关的入口、模块、上下游依赖和必要测试/配置线索
4. 追踪关键调用链、数据链路或状态链路
5. 将判断整理成“高置信原因”与“待确认假设”
6. 在目标仓库里写入新的历史分析快照，并刷新 `current-review.md`
7. 在对话里给摘要，并明确用户可以：
   - 补充信息
   - 直接修改文档后让 agent 继续修订
   - 明确同意进入 `code-plan`
8. 如果用户反馈了新信息，重新读取最新 `current-review.md` 和用户新增证据，生成新的分析快照并刷新稳定入口
9. 如果用户明确同意进入修改方案设计，则切换到 `code-plan`

工作流必须保持 symptom-first，而不是偷偷退化成 diff-first 或 repo-wide analysis。

## 报告结构

每次正式分析文档都必须包含以下部分，且顺序固定：

1. `## 审查范围`
2. `## Bug 表现与输入上下文`
3. `## 当前已确认事实`
4. `## 相关模块与关键链路`
5. `## 高置信原因判断`
6. `## 待确认假设与缺失证据`
7. `## 建议补充的验证动作`
8. `## 用户反馈与修订记录`
9. `## 后续入口`
10. `## 未覆盖范围`

### 各部分说明

#### `## 审查范围`

说明当前分析覆盖了哪些模块、文件、入口、调用链或配置面，避免用户把结果误读成仓库级结论。

#### `## Bug 表现与输入上下文`

原样整理用户提供的 bug 表现、复现线索、环境差异、错误信息或怀疑点。这里是分析的起点，不应在没有证据时替用户改写成“已经成立的结论”。

#### `## 当前已确认事实`

只记录已被当前仓库代码、配置、测试、日志片段或用户补充证据证实的事实。不要把猜测和事实混写。

#### `## 相关模块与关键链路`

把与当前 bug 最相关的模块关系和链路写清楚。必要时可以按“入口 -> 中间层 -> 下游依赖 / 状态流转”的顺序描述。

#### `## 高置信原因判断`

每项都要包含：

- 简短标题
- 对应的现象对齐证据
- 对应的源码证据
- 为什么这个原因能解释当前 bug
- 修复时大概率会涉及的模块方向

这里仍然是分析结论，不是正式修改方案。

#### `## 待确认假设与缺失证据`

列出当前仍需人工补充或继续验证的怀疑点，避免把分析写成“结案报告”。

#### `## 建议补充的验证动作`

只给与当前假设直接相关、最小但有判别力的验证动作，例如回看某段调用、补一条日志、人工复现某个分支，或确认某个配置值。

#### `## 用户反馈与修订记录`

记录本轮用户新补充了什么、纠正了什么、文档相对上一轮更新了哪些结论。这样后续读者能理解分析是如何演进的。

#### `## 后续入口`

明确给出两类后续路径：

- 继续补充和修订 bug 分析
- 在用户确认后进入 `code-plan`

#### `## 未覆盖范围`

说明当前没有展开审查的相邻模块或分支，避免用户误以为所有边界都已经被覆盖。

## 文档修订模型

`debug-review` 不是一次性产物。它必须支持两种修订来源：

### 对话内反馈

如果用户直接在对话里补充现象、订正理解、指出误判、追加路径、贴出日志或说明某个结论不成立，agent 应把这些内容视为当前分析的增量输入。

### 文档内反馈

如果用户直接修改了 `debug-review-docs/current-review.md` 或最新历史分析文档，agent 应把这些修改视为显式人工订正，而不是覆盖掉用户意图。

### 修订规则

- 先读取最新 `current-review.md`
- 再读取最新一份历史分析快照
- 对照用户新增反馈，决定哪些结论应保留、降级、删除或补强
- 写出新的历史分析快照
- 刷新 `current-review.md`

这样能保证：

- 当前文档始终是最新可工作的版本
- 每一轮重要修订都保留历史证据链

## 与 `code-plan` 的衔接

### 默认行为

默认停在 bug 分析阶段。即使已经给出高置信原因，也不自动生成修改方案。

### 进入条件

只有当用户明确表达“同意进入修改方案设计”时，才从当前线程切换到 `code-plan`。例如：

- “继续出修改方案”
- “按这份分析进入 code-plan”
- “可以开始设计修复方案了”

### handoff 上下文

切换到 `code-plan` 时，应把以下内容视为上游上下文：

- `debug-review-docs/current-review.md`
- 最新一份 `debug-review-docs/reviews/YYYY-MM-DD-HHmm-<topic>.md`
- 用户在对话里最后确认的补充信息

### 对 `code-plan` 的契约补充

为了让 handoff 自然成立，`code-plan` 的触发规则需要从“仅显式 `/code-plan`”扩成：

- 用户显式输入 `/code-plan`
- 或者当前线程已经在 `debug-review` 工作流中，且用户明确同意进入修改方案设计

这个补充必须保持窄边界：它只接受来自 `debug-review` 的已确认 handoff，不接受普通 bug 对话的隐式跳转。

## 与现有 skill 的边界

### 相对 `code-review`

`code-review` 是 diff-first。`debug-review` 是 symptom-first。前者围绕当前改动找遗漏，后者围绕 bug 现象找原因。

### 相对 `code-analyse`

`code-analyse` 适合长期、体系化的模块或仓库说明。`debug-review` 只围绕一个 bug 任务写 task-scoped 分析，不应该扩张成长期架构文档。

### 相对 `skill-debug`

`skill-debug` 关注 skill 本身的 trace/debug 接入。`debug-review` 关注业务项目里的 bug 原因分析。这两个能力可以并存，但职责不同。

### 相对 `code-plan`

`debug-review` 给出“为什么坏了”的分析文档；`code-plan` 负责在用户确认后，把“准备怎么改”整理成实现前 spec。

## 建议目录结构

```text
debug-review/
  SKILL.md
  README.md
  USAGE.md
  templates/
    review-report.md
    current-review.md
  references/
    analysis-checklist.md
    evidence-rules.md
    handoff-to-code-plan.md
  evals/
    evals.json
```

说明：

- `analysis-checklist.md`：沉淀 symptom-first 的分析清单
- `evidence-rules.md`：沉淀高置信 / 假设分层规则
- `handoff-to-code-plan.md`：明确何时允许交给 `code-plan`

这个结构比引入大量专项 pack 更克制，先把基础工作流做稳，再决定是否扩成 pack 化。

## README 与分发要求

`README.md` 至少要覆盖：

- skill 的定位与边界
- 典型触发方式 `/debug-review`
- 默认输出目录 `debug-review-docs/`
- 用户如何通过对话或直接改文档来修订分析
- 用户如何在确认后进入 `code-plan`
- 目录版安装方式
- 打包版 `.skill` 安装方式
- 复制到其他工作区的方式

同时应生成：

- `dist/debug-review.skill`

以便和现有 skill 的分发方式保持一致。

## 评测建议

初始 `evals/evals.json` 至少覆盖三类 prompt：

1. 用户描述一个具体 bug 现象，请求 agent 审查当前项目相关模块并生成分析文档
2. 用户在首轮分析后补充新线索，要求 agent 修订 bug 分析文档
3. 用户认可分析并要求进入修改方案设计，验证是否能自然 handoff 到 `code-plan`

这三类样例能覆盖该 skill 的核心价值，而不会一上来就把评测做得过宽。

## 已确定决策

- skill 名称：`debug-review`
- 触发命令：`/debug-review`
- 默认目标：当前工作仓库
- 默认输出目录：`<target-project>/debug-review-docs/`
- 文档模型：`current-review.md` + `reviews/YYYY-MM-DD-HHmm-<topic>.md`
- 交互模型：先 bug 分析，后续支持修订
- handoff 规则：只有用户明确同意进入修改方案设计时，才切换到 `code-plan`
