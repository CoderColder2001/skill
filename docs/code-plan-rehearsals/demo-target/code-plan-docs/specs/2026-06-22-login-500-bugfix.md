# 登录接口 500 Bugfix 预审 Spec

## 当前编码意图

- 当前任务是在现有 `demo-target` 项目里修复登录接口偶发返回 `500` 的问题，而不是重构完整认证系统。
- 这次改动想要改变的行为是：当用户不存在或用户数据不完整时，登录链路不再因为未处理的空值访问而抛出异常。
- 这次改动现在需要做，是因为当前 `login` 主流程默认仓储层一定返回用户对象，这和 `findUserByEmail()` 的返回契约不一致，已经能从测试里看出会触发异常路径。

## 预期目标与完成标准

- 当 `findUserByEmail()` 返回 `null` 时，登录请求不再抛出未处理异常。
- 登录失败路径返回可预期的失败结果，而不是把底层异常直接暴露到路由层。
- `loginRoute()` 不再把所有失败都映射成 `500`，至少要为可预期的认证失败保留更合理的响应空间。
- 回归测试覆盖“用户不存在”场景，并验证修复后行为符合预期。
- 正常登录成功路径保持现有成功响应不变。

## 当前相关模块现状

- `src/routes/auth.ts` 通过 `login()` 执行登录主流程，并且当前只根据 `result.ok` 决定成功或失败响应。失败时统一返回 `500` 和 `LOGIN_FAILED`。
- `src/services/auth-service.ts` 负责读取用户、校验密码并生成 token。这里当前直接访问 `user.passwordHash`，默认 `findUserByEmail()` 必然返回用户对象。
- `src/repositories/user-repo.ts` 的返回类型明确是 `Promise<UserRecord | null>`，说明“用户不存在”本来就是允许返回的正常分支，而不是异常场景。
- `tests/auth/login.test.ts` 已经说明现状：当查询缺失用户时，`loginRoute()` 会抛异常，而不是返回受控失败结果。

## 本次涉及改动模块

- `src/services/auth-service.ts`
  - 变更原因：这里是空值防御缺失的根源位置，需要补足 `null` 分支处理和失败结果表达。
  - 预期职责变化：继续保留登录主流程编排职责，但补齐返回契约与仓储层返回类型之间的防御逻辑。
- `src/routes/auth.ts`
  - 变更原因：当前把所有失败统一映射为 `500`，会放大业务失败与系统错误之间的界线不清。
  - 预期职责变化：仍然只负责 HTTP 响应组装，但要按新的失败语义选择更合适的状态码或错误体。
- `tests/auth/login.test.ts`
  - 变更原因：需要把“当前会抛异常”的测试更新成回归测试，验证修复后的受控失败行为。
  - 预期职责变化：增加对缺失用户场景的断言，不再只验证异常抛出。
- `src/repositories/user-repo.ts`
  - 变更原因：主要用于确认现有返回契约，不一定需要代码修改。
  - 预期职责变化：原则上不改业务职责；如有需要，也只做轻量注释或类型澄清。

## 新模块架构设计陈述

- 本次不计划引入新模块。
- 原因是问题集中在现有登录链路的契约处理和错误映射，边界已经相对清晰。新增模块会让一个局部 bugfix 变成额外的结构迁移，收益不高。

## 风险与待确认项

- 需要确认登录失败场景在这个项目里期望返回什么状态码；如果统一保留 `500`，那只能修复崩溃而无法真正改善错误语义。
- 如果后续打算区分“用户不存在”“密码错误”“系统异常”，`auth-service.ts` 的失败结果结构可能需要进一步细化。
- 当前测试只覆盖成功场景和异常抛出场景，历史覆盖较弱，改动时要防止只修了一个分支却漏掉其他失败路径。

## 不在本次范围内的内容

- 不重写认证服务的数据结构。
- 不引入新的鉴权中间件或 session 方案。
- 不顺手重构整个错误码体系。
