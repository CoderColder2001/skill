# Evidence Rules

`debug-review` 的输出必须把“已经能站住的判断”和“还需要验证的猜测”拆开。

## 高置信原因

只有满足下面两个条件时，才能进入高置信原因：

1. **现象对齐证据**
   - traced path、状态变化或条件分支，能够解释用户看到的 bug 表现
2. **源码证据**
   - 当前仓库里的代码、配置、测试、静态资源、构建配置或用户给出的明确证据支撑这个判断

写高置信原因时，至少交代：

- 哪个现象被解释了
- 哪段代码或链路支撑了判断
- 为什么这不是一个“也许”

## 待确认假设

只要下面任一条件缺失，就留在待确认假设：

- 还不能解释 bug 表现
- 还没有找到代码或配置证据
- 只能靠项目习惯、猜测或经验推断

每个假设都要说明：

- 为什么值得继续查
- 缺了什么证据
- 最小验证动作是什么

## 修订时的降级规则

如果用户补充的新信息推翻了一个原先的高置信原因：

- 不要硬保留原结论
- 先把它降级到待确认假设，或者直接移除
- 在“用户反馈与修订记录”里写清为什么变了

## 写法提醒

- “可能”不是问题，混写才是问题
- 用清晰分层降低噪音，而不是用更强语气掩盖证据不足
- 不要把“建议验证动作”写成“已验证事实”
