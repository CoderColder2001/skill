# Symptom-First Analysis Checklist

在 `/debug-review` 里，先按下面顺序收敛分析：

1. 先把 bug 表现、复现线索、环境差异和用户怀疑点拆开记录，不要一上来就把它们写成“根因”。
2. 找最近的入口点：页面动作、接口、命令、job、worker、配置分支，先从最接近症状的触发点开始。
3. 只追对当前 bug 真有判别力的上下游链路：
   - 调用方
   - 被调方
   - 数据转换
   - 状态流转
   - 条件分支
4. 只有当配置、测试、feature flag、fallback 分支会改变当前解释时，才把它们拉进分析范围。
5. 如果证据不足，就停在“待确认假设”，不要用更长的文档把猜测写得像事实。
6. 分析结束后，明确告诉用户：
   - 当前高置信原因是什么
   - 还缺什么证据
   - 用户可以如何补充或修订

避免这些偏航：

- 偷偷切成 diff review
- 偷偷扩成仓库级架构文档
- 因为用户说了一个怀疑点，就把它当成结论
- 用“建议加日志”代替真正的链路解释
