# 登录接口 500 Bugfix 预审 Spec

## 当前编码意图

- 用户当前希望修复登录接口偶发返回 `500` 的问题，而不是重做整套认证链路。
- 这次改动想要改变的行为：当上游用户状态读取失败或返回空值时，接口应给出可预期的业务响应，不再直接抛出未处理异常。
- 这次改动为什么现在需要做：当前线上问题已经影响登录成功率，优先级高，而且错误范围集中在现有登录链路内，适合先做定点修复。

## 预期目标与完成标准

- 登录接口在已知异常输入和上游空返回场景下不再返回 `500`。
- 接口错误分支返回符合现有 API 契约的状态码和错误体。
- 相关单元测试或集成测试覆盖本次修复的核心分支。
- 不改变正常登录成功路径的现有行为。

## 当前相关模块现状

- `src/routes/auth.ts` 负责接收登录请求并把参数转交给认证服务。
- `src/services/auth-service.ts` 当前封装登录主流程，包括读取用户、校验密码、生成 token。
- `src/repositories/user-repo.ts` 负责从存储层读取用户资料，当前调用方默认它总能返回结构完整的用户对象。
- 现状里的主要问题是认证服务对空返回和异常分支的防御不足，导致底层异常直接冒泡到路由层。

## 本次涉及改动模块

- `src/routes/auth.ts`
  - 变更原因：需要确认路由层当前错误映射是否会把业务异常统一翻成 `500`。
  - 预期职责变化：尽量不扩大职责，只调整与本次 bugfix 直接相关的错误响应映射。
- `src/services/auth-service.ts`
  - 变更原因：这里是本次异常防御和业务分支补齐的核心位置。
  - 预期职责变化：补足空返回、异常包装和可预期失败分支，但不顺手重写认证流程。
- `src/repositories/user-repo.ts`
  - 变更原因：需要确认当前返回契约是否稳定，以及调用方应如何处理空值。
  - 预期职责变化：如果现有返回契约不清晰，可能只补充注释、类型或轻量保护，不扩大成仓储层重构。
- `tests/auth/login.test.ts`
  - 变更原因：需要补覆盖这次 bugfix 的回归测试。
  - 预期职责变化：增加异常路径和空返回路径测试。

## 新模块架构设计陈述

- 本次不计划引入新模块。
- 原因：问题集中在现有登录调用链的异常处理与契约防御，当前边界仍然清晰，新增模块只会让一个定点 bugfix 变成不必要的结构调整。

## 风险与待确认项

- 需要确认前端或调用方是否依赖当前错误体结构，否则修复时可能引入兼容性问题。
- 需要确认 `user-repo` 在不同存储后端下是否都可能返回空值或结构不完整数据。
- 如果历史测试覆盖较弱，可能需要先补最小回归测试才能安全收口修复。

## 不在本次范围内的内容

- 不重构整个认证架构。
- 不引入新的认证 provider 抽象层。
- 不顺手整理所有历史错误码。
